Performance issues

Critical vendor-core.js is causing 18 long tasks (2.1 s total execution)

Your vendor-core-Dn8JRaNM.js (347 KB) is responsible for most of the 18 long tasks blocking the main thread — 4 tasks of 160–227 ms each all stem from it. This is your bundled React + UI library chunk. It's parsed and executed synchronously, freezing interaction for nearly 9 seconds until TTI.

Split vendor-core further in vite.config.js — separate React itself from your UI component library.
Use dynamic imports for heavy components: const Player = lazy(() => import('./Player')).
Defer non-critical UI (sidebar, queue) until after initial paint using Suspense boundaries.

Critical Google APIs (gapi) blocking for 269 ms — worst single long task

The longest single long task (269 ms at 10,379 ms after load start) is from apis.google.com/gapi_iframes — Google's auth iframe script. This is Firebase Authentication loading its cross-origin iframe. It's being loaded synchronously and blocking interaction late into page load.

Defer Firebase Auth initialization — don't call onAuthStateChanged on app start; wait until user interacts.
Load Firebase Auth lazily: const auth = await import('firebase/auth') only when login is needed.
Your vendor-firebase-B-BH3HQM.js is 503 KB — split auth from other Firebase services (Firestore, Storage) using separate dynamic imports.
Critical LCP image not discoverable — resource load delay 605 ms

The LCP element is an album art image from c.saavncdn.com — the vinyl label image in your Now Spinning screen. It fails both key checks: it's not in the initial HTML (injected by React after JS runs), and it has no fetchpriority=high. The browser only discovers it after parsing and running your JS bundles.

Add fetchpriority="high" to your primary album art <img> tag — this alone will cut LCP significantly.
Add <link rel="preconnect" href="https://c.saavncdn.com"> in your HTML <head> so the connection is ready before JS runs.
If you know the current track on load, preload its image: <link rel="preload" as="image" href="{albumArtUrl}">.
High Render-blocking: Google Fonts + CSS blocking paint for ~839 ms Est. save 300 ms

Two resources block the browser from painting anything: the Space Mono font request to fonts.googleapis.com and your main CSS file. The browser must finish downloading both before it can show any content — this directly causes your 2.6 s FCP.

Add rel="preconnect" for Google Fonts and load it with media="print" onload="this.media='all'" to make it non-blocking.
Or self-host Space Mono using fontsource: npm install @fontsource/space-mono and import it directly.
Add <link rel="preload" as="style"> for your CSS in index.html so it loads in parallel with JS.
High JioSaavn API has 770 ms server latency

Your backend jiosavnapi-production.up.railway.app takes 770 ms to respond — nearly 1 second just waiting for data before anything plays. This is a Railway.app cold start issue (free/hobby tier spins down after inactivity) combined with the API being a single-region server.

Upgrade Railway to a paid plan or use an always-on service to prevent cold starts.
Cache API responses in Firestore or localStorage — on revisit, show cached songs instantly while refreshing in background.
Add a loading skeleton immediately on mount so the 770 ms feels shorter — perceived performance matters.
High Forced reflow in index.js — 81 ms wasted per occurrence

Your code at index.js line 4 col 6099 and line 3 col 6326 is reading layout properties (like offsetHeight, scrollTop, getBoundingClientRect) immediately after writing to the DOM. This forces the browser to recalculate the entire layout synchronously — 81 ms per hit, adding to Style & Layout's 1.27 s total.

Read all layout values first, then batch all DOM writes — never interleave reads and writes.
Use requestAnimationFrame to defer DOM measurements to after paint.
Common culprit in music apps: scrolling the now-playing item into view — wrap in setTimeout(fn, 0).

Cache and image issues

High Your own JS bundles have only 1-hour cache — waste 482 KiB on every revisit Est. save 482 KiB

Firebase Hosting defaults to 1-hour cache (Cache-Control: max-age=3600) for your JS/CSS assets. Since your filenames are already content-hashed (e.g. vendor-core-Dn8JRaNM.js), they're safe to cache for 1 year — but Firebase doesn't know that automatically. Saavn album art images have 0-second cache, re-downloading on every page view.

Add this to your firebase.json:
"hosting": {
  "headers": [{
    "source": "/assets/**",
    "headers": [{ "key": "Cache-Control",
      "value": "public,max-age=31536000,immutable" }]
  }]
}
For album art from saavncdn.com: cache responses in IndexedDB or localStorage with a 24h TTL on your side.

Album art from saavncdn.com is served as 500×500 JPEG but displayed at ~143×143 px — you're downloading ~12× more pixels than needed. The images are also not being served in WebP/AVIF format despite your app requesting them at display size.

JioSaavn CDN URLs support size params — request 150x150 instead of 500x500 by replacing the size suffix in the URL string.
Add loading="lazy" to all album art not in the initial viewport — only the now-playing image should be eager.

Accessibility issues (also affect SEO score)

Info Buttons missing accessible names + low contrast text

Multiple icon-only buttons (play, skip, like) have no aria-label. Screen readers announce them as "button" with no context. Color contrast on some text elements is also below WCAG AA — likely light gray text on your dark surface.

Add aria-label to every icon button: <button aria-label="Play"><PlayIcon /></button>.
Check contrast of secondary text (artist names, durations) against your background — target 4.5:1 ratio minimum.
Fix heading order — an <h4> appears before any <h2> or <h3> in your sidebar.
